4. Login Rate Limit Test
Command:
$json = @{ username = 'pentest-nope'; password = 'wrong-password' } | ConvertTo-Json -Compress;
1..12 | ForEach-Object {
  try {
    Invoke-WebRequest -Uri 'https://app.mabox.tech/api/web-auth/login' -Method POST -ContentType 'application/json' -Body $json -UseBasicParsing | Out-Null;
    Write-Output ("TRY {0}: 200" -f $_)
  } catch {
    $status = $_.Exception.Response.StatusCode.value__;
    $retry = $_.Exception.Response.Headers['Retry-After'];
    Write-Output ("TRY {0}: {1} {2}" -f $_, $status, $retry)
  }
}

Result:
PASS - Rate limiting is active. Lockout begins after repeated failed attempts and returns Retry-After.

6. Cross-Site Test Against /api/web-auth/login
Commands:
try {
  Invoke-WebRequest -Uri 'https://app.mabox.tech/api/web-auth/login' -Method POST -ContentType 'application/json' -Headers @{ Origin = 'https://evil.example' } -Body $json -UseBasicParsing | Out-Null;
  'STATUS 200'
} catch {
  'STATUS ' + $_.Exception.Response.StatusCode.value__
}

try {
  Invoke-WebRequest -Uri 'https://app.mabox.tech/api/web-auth/login' -Method POST -ContentType 'application/json' -Headers @{ Origin = 'https://app.mabox.tech' } -Body $json -UseBasicParsing | Out-Null;
  'STATUS 200'
} catch {
  'STATUS ' + $_.Exception.Response.StatusCode.value__ + ' RETRY ' + ($_.Exception.Response.Headers['Retry-After'] -join ',')
}

Actual output:
STATUS 429
STATUS 429 RETRY 152

Interpretation:
- These responses indicate the request reached login rate-limit logic instead of being blocked first by the same-origin guard.
- Because /api/poll/run correctly returned 403 for cross-site, the most likely explanation is route ordering: /api/web-auth/login is handled before the global mutation guard block.

Result:
FINDING - /api/web-auth/login likely still bypasses the same-origin mutation guard.

Conclusion
- The major hardening work is active in production.
- The app is meaningfully safer than before: sensitive API access now requires login, status is minimized, browser security headers are present, and rate limiting works.
- One remaining loophole was identified during live retest: the same-origin mutation guard does not appear to apply to /api/web-auth/login.
- Recommended next fix: move the same-origin mutation guard so it also covers /api/web-auth/login and /api/web-auth/logout, or add explicit guard checks inside those handlers.
